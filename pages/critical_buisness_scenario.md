## 7) Критические бизнес-сценарии

### Сценарий 1: Идентификация существующего покупателя
- **Описание**: Пользователь, уже совершавший покупки на сайте компании, регистрируется в приложении. Система должна связать его спортивный профиль с историей покупок, чтобы начать персонализированный цикл взаимодействия.
- **Цель**: Превратить анонимного пользователя приложения в известного клиента компании. Связать его спортивную активность с историей покупок.
- **Действия**: Регистрация через email (использованный на сайте) → Предложение привязать прошлые заказы → Импорт купленной экипировки в цифровой профиль.
- **Входные данные**: Email/телефон (при регистрации), данные OAuth-провайдера (Google, Apple).
- **Выходные данные**: Созданный/найденный профиль пользователя, список купленных товаров, их артикулы, даты покупок.
- **Задействованные компоненты**:
- - Сервис аутентификации
- - Сервис профилей
- - Каталог
- - Сервис инвентаря


### Сценарий 2: Автоматический учёт износа экипировки и триггер для покупки
- **Описание**: После каждой тренировки система пересчитывает степень износа используемой экипировки. При достижении порога износа пользователю отправляется персонализированное предложение о замене.

- **Цель**: Перевести абстрактный износ в конкретное, персонализированное уведомление-предложение.
- **Действия**: Пользователь добавляет инвентарь в профиль → Система автоматически рассчитывает износ по тренировкам → При достижении порога, пользователь получает уведомление.
- **Входные данные**: Завершенная тренировка (тип активности, метрики: дистанция, время), информация об использованной в тренировке экипировке (ID из профиля).
- **Выходные данные**:
- - Обновленный показатель износа в профиле экипировки.
- - Push-уведомление/SMS-сообщение с рекомендацией замены.
- - Deep-link на страницу товара в магазине компании (той же или новой модели).
- **Задействованные компоненты**:
- - Сервис тренировок
- - Сервис инвентаря
- - Сервис рекомендаций
- - Сервис уведомлений


### Сценарий 3: «Соревновательный азарт» и социальное доказательство
- **Описание**: Пользователь, участвуя в челлендже и видя успехи друзей или лидерборды, испытывает желание улучшить свои результаты. Система использует этот момент для предложения товаров, которые могут помочь (например, более легкая обувь для бега, трекер пульса).
- **Цель**: Использовать механику групп и челленджей для создания моды на новые продукты.
- **Входные данные**:
Действие пользователя: просмотр лидерборда, вступление в челлендж, сравнение своих результатов с результатом друга.
Контекст: вид спорта, текущая экипировка пользователя, его исторические данные.
- **Выходные данные**: Контекстная реклама/рекомендация товара, интегрированная в социальный/игровой интерфейс (например, баннер «Экипировка лидеров» в разделе челленджа, сравнение: «Твой друг использует кроссовки модели X»).
- **Задействованные компоненты**:
- - Сервис геймификации
- - Сервис рекомендаций
- - Сервис контента

### Сценарий 4: Рекомендация товаров-комплементов на основе тренировочного плана
- **Описание**: Пользователь начинает новый тренировочный план (например, «Подготовка к марафону»). Система анализирует план, текущую экипировку пользователя и рекомендует товары-комплементы (гель для восстановления, новые носки, гидропак, книга по подготовке).
- **Цель**: Увеличение среднего чека за счет кросс-продаж и предложения решения «под ключ».
- **Входные данные**: Выбранный пользователем тренировочный план, текущий инвентарь, сезон, локация.
- **Выходные данные**: Блок «Рекомендуемая экипировка для этого плана» с товарами, которых нет у пользователя.
- **Задействованные компоненты**:
- - Сервис тренировок
- - Сервис инвентаря
- - Сервис рекомендаций
- - Каталог
